# Phase 10-01: Custom Validation Framework

## Objective

Implement a custom validation rule framework that allows users to register and execute domain-specific validation rules on Diode devices. This enables users to enforce business logic that goes beyond basic schema validation (e.g., "device role must match device type capabilities", "serial number format per manufacturer").

**Purpose:** Enable extensible validation logic without modifying the core library
**Output:** Validation framework with Rule base class, RuleRegistry, and ValidatorPipeline

# Framework Architecture

## Core Components

### ValidationRule (Base Class)
Abstract base class for all validation rules. Provides:
- `name`: Unique identifier for the rule
- `description`: Human-readable description
- `apply(device)`: Abstract method to implement in subclasses
- `get_severity()`: Rule severity (error, warning, info)

### RuleRegistry
Global registry that stores registered rules and provides:
- `register(rule)`: Add a new rule
- `get(rule_name)`: Retrieve a rule by name
- `get_all()`: Get all registered rules
- `unregister(rule_name)`: Remove a rule

### ValidatorPipeline
Orchestrates validation execution:
- `add_rule(rule)`: Add a rule to the pipeline
- `set_severity_filter(severity)`: Filter rules by severity
- `run(device)`: Execute all rules against a device
- `get_results()`: Get validation results

## Validation Result Structure

```python
@dataclass
class ValidationResult:
    rule_name: str
    rule_description: str
    severity: str  # "error", "warning", "info"
    passed: bool
    field: Optional[str] = None
    value: Any = None
    error_message: Optional[str] = None
    metadata: dict = field(default_factory=dict)
```

## Severity Levels

| Severity | Description | Impact on Device |
|----------|-------------|------------------|
| `error` | Validation fails, device should not be sent to Diode | Blocks transmission |
| `warning` | Issue detected but device may still be valid | Logs warning |
| `info` | Informational message, no validation issue | Logs for debugging |

## Built-in Rules

| Rule Name | Description | Severity | Field(s) |
|-----------|-------------|----------|----------|
| `required_fields` | Check all required fields are present | error | name, site, device_type, role |
| `valid_status` | Validate status is one of allowed values | error | status |
| `name_length` | Validate name length (1-64 chars) | error | name |
| `serial_length` | Validate serial length (0-64 chars) | warning | serial |
| `asset_tag_length` | Validate asset_tag length (0-64 chars) | warning | asset_tag |
| `role_match_device_type` | Validate role is appropriate for device type | warning | role, device_type |

## Security Considerations

| Boundary | Threat | Mitigation |
|----------|--------|------------|
| Rule Registration | Unauthorized rule injection | Rule name validation, type checking |
| Rule Execution | Malicious rule causing DoS | Timeout support, severity filtering |
| Data Validation | Bypassing validation via custom rules | Built-in rules run first |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-10-01-01 | Tampering | RuleRegistry.register() | mitigate | Validate rule name format, check for duplicates, enforce type Safety |
| T-10-01-02 | Spoofing | ValidationRule.apply() | mitigate | Type hints, runtime type checking in `apply()` method |
| T-10-01-03 | Information Disclosure | ValidationResult | accept | Results contain only field names and values, no secrets |
| T-10-01-04 | Denial of Service | ValidatorPipeline.run() | mitigate | Add timeout parameter, limit rule execution time |
| T-10-01-05 | Elevation of Privilege | Rule Priority | mitigate | Built-in rules always run first, custom rules run after |
