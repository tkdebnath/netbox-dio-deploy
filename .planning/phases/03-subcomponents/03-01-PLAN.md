# Phase 03: Device Subcomponents - PLAN 01

**Implement Pydantic models and conversion functions for device subcomponents**

## Objective

Create a complete set of Pydantic models for device subcomponents (Interfaces, VLANs, Modules, Cables, Prefixes, IP Addresses) with conversion functions to transform them into Diode Entity protobuf messages for gRPC transmission to NetBox Diode.

**Purpose:** Enable network automation engineers to define device subcomponents as Python dictionaries and push them to NetBox Diode with minimal code, while enforcing data integrity and validation through Pydantic.

**Output:** Full set of Pydantic models with validation, conversion functions, and comprehensive test coverage for all subcomponent types.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Client→Converter | Untrusted Pydantic model data (from user dictionary) crosses here |
| Converter→Diode SDK | Converted protobuf data crosses into gRPC client |
| User→Model | User-provided dictionary data enters Pydantic model validation |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-01 | S | All Pydantic models | mitigate | Pydantic model_validate() enforces type safety at construction time |
| T-03-02 | I | Converter functions | mitigate | Validate input is correct Pydantic model type; raise ValueError for incorrect types |
| T-03-03 | T | Field mapping | mitigate | Explicit field-to-field mapping in to_protobuf() prevents data corruption |
| T-03-04 | T | Enum validation | mitigate | Pydantic field validators ensure enum values are from allowed set |
| T-03-05 | D | Nested object handling | mitigate | All nested objects convert to separate Entities; no assumption about nesting order |
| T-03-06 | I | Cable termination points | mitigate | Validate termination_type and cable_end fields; raise ValueError for invalid combinations |
| T-03-07 | T | Prefix validation | mitigate | Use regex or ipaddress module to validate prefix format (e.g., 192.168.1.0/24) |

</threat_model>
