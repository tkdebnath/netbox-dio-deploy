Create a robust error handling system for the netbox-dio package that provides clear, actionable error messages at every layer of the application. This phase addresses the requirements for:
1. Invalid data validation with descriptive error messages

3. Conversion errors with source data context

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| User Input -> DiodeDevice | Untrusted dictionary data enters validation layer |
| Pydantic -> Diode SDK | Converted data crosses into external system |
| Client -> gRPC Server | Network transmission with potential failure modes |
| Batch -> Individual Devices | Batch processing must isolate failures to prevent cascade |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-05-01 | S | Exception messages | mitigate | Sanitize error messages; never include sensitive data; provide actionable guidance |
| T-05-02 | T | Device validation | mitigate | Validate all required fields before processing; raise DiodeValidationError with field names |
| T-05-03 | I | Pydantic ValidationError | mitigate | Catch and re-raise as DiodeValidationError with context (field name, value, device name) |
| T-05-04 | R | gRPC connection failures | mitigate | Distinguish connection refused, timeout, auth errors with specific actionable messages |
| T-05-05 | D | Batch processing | mitigate | Continue processing remaining devices after failures; report per-device errors |
| T-05-06 | E | Conversion errors | mitigate | Include device name and context in DiodeConversionError; preserve traceback for debugging |
| T-05-07 | I | Config validation | mitigate | Validate endpoint format, cert file existence; raise DiodeValidationError with clear instructions |
| T-05-08 | D | Error aggregation | mitigate | Aggregate all errors in BatchResult; provide summary statistics for debugging |

## Error Message Guidelines

All exception messages should follow these principles:
1. **What failed**: Clear description of the failure
2. **Where**: Include device name, field name, or operation
3. **Why**: Reason for the failure
4. **How**: Suggested fix when possible

Example formats:
- `Device '{name}': {field_name} validation failed: {reason}. Expected: {expected}.`
- `Failed to connect to Diode at {endpoint}: {reason}. Check your network configuration.`
- `Conversion error on device '{name}': {reason}. Check source data format.`
